# Хранение файлов в адаптерах ioBroker
В этом документе объясняется, как правильно сохранять файлы в базе данных ioBroker с использованием `writeFileAsync`. Он основан на обсуждениях разработчиков, но написан в **нейтральном формате, напоминающем документацию**, с примерами.

---

## Обзор
Адаптеры ioBroker могут хранить файлы во внутренней базе данных. Для корректной работы этой функции необходимо определить **метаобъекты**, которые выступают в качестве пространств имен хранилища. Выбор метаобъекта определяет, будут ли файлы включены в резервные копии или нет.

---

## Метаобъекты
Существует два соответствующих типа метаобъектов:

- **`мета`**
- Файлы, хранящиеся здесь, **не включаются в резервные копии**.
- Подходит для временных или восстанавливаемых данных.

- **`meta.user`**
- Файлы, хранящиеся здесь, **входят в состав резервных копий**.
- Подходит для хранения постоянных пользовательских данных, таких как ключи, файлы конфигурации или загруженный контент.

Метаобъект действует как **точка монтирования** для файлов.
Затем команда `writeFileAsync` записывает данные относительно этого базового пути.

---

## Определение метаобъекта
Перед записью файлов создайте метаобъект. Пример:

```json
{
  "_id": "keys",
  "type": "meta",
  "common": {
    "name": "keys",
    "role": "meta.user"
  },
  "native": {}
}
```

Здесь:

- `_id`: определяет пространство имен хранилища (в данном случае `keys`).
- `type: "meta"`: требуется для объектов хранения.
- `role: "meta.user"`: гарантирует включение данных в резервные копии.

---

## Запись файлов
После создания метаобъекта запись в файлы может осуществляться с помощью `writeFileAsync`.

### Пример: Написание закрытого ключа
```js
// Store private key in namespace "adapter.namespace.keys"
await adapter.writeFileAsync(
  `${adapter.namespace}.keys`,    // meta object mount point
  'private-key.pem',              // relative file path
  keys.privateKey                 // file content
);
```

### Пример: Создание временного файла
```js
// Store temporary data in namespace "adapter.namespace.temp"
await adapter.writeFileAsync(
  `${adapter.namespace}.temp`,    // meta object mount point
  'cache.json',                   // relative file path
  JSON.stringify(cacheData)       // file content
);
```

---

## Практические заметки
- Без метаобъекта вызов функции `writeFileAsync` завершится ошибкой.
— Всегда выбирайте между `meta` и `meta.user`:
- Используйте **`meta`**, если содержимое можно восстановить и его не следует резервировать.
— Используйте **`meta.user`** для постоянных файлов, связанных с пользователем, которые должны сохраниться после резервного копирования.
- Первым параметром в `writeFileAsync` является метаобъект (пространство имен хранилища).
- **Второй параметр** — это путь относительно этого пространства имен.

---

## Пример от ioBroker Sayit
В разделе [адаптер Sayit](https://github.com/ioBroker/ioBroker.sayit) определены два пространства имен хранения в разделе `io-package.json`:

1. `adapter.namespace` (временное хранилище, не резервное копирование)
2. Хранилище `meta.user` (постоянное, с резервным копированием)

Этот подход позволяет адаптеру отделять временно сгенерированные файлы от контента, предоставленного пользователем.

---

## Краткое содержание
- Используйте **метаобъекты** для определения пространств имен хранилища в ioBroker.
- Выберите один из вариантов:
- `meta` → временный, исключается из резервных копий.
- `meta.user` → постоянный, включается в резервные копии.
- Рассматривайте метаобъекты как **точки монтирования**.
- Всегда проходите:
- **первый параметр** = пространство имен (например, `adapter.namespace.keys`).
- **второй параметр** = относительный путь к файлу.

Правильное использование гарантирует надежное хранение файлов и их включение в резервные копии при необходимости.
