# IoBroker.sunenergyxt500
**Тесты:** ![Тестирование и выпуск](https://github.com/Creekhail/ioBroker.sunenergyxt500/workflows/Test%20and%20Release/badge.svg)

## Адаптер sunenergyxt500 для ioBroker
Интеграция и управление собственным потреблением для **[SunEnergyXT 500 / 500 PRO](https://www.sunenergyxt.com/details-500-series)** Гибридные системы хранения энергии с питанием от сети переменного тока (производитель: SunEnergyXT)](https://www.sunenergyxt.com/)) через **локальный HTTP API** устройства — облачная учетная запись не требуется. Один экземпляр управляет **до тремя головками** (башнями хранения).

## Функции
* Управляет от **одной до тремя головами** в одном экземпляре, каждая из которых находится в собственном поддереве `heads.<n>.*`, а также объединенными агрегатами `total.*`.
* Опрашивает локальный API (`GET /read`) и отображает все стабильные поля в состояниях: уровень заряда батареи (SoC), мощность батареи/сети/нагрузки/солнечной батареи, ток/напряжение для каждого MPPT, ежедневные счетчики энергии, уровень заряда батареи (SoC) для каждого блока, информация об устройстве/прошивке и состояние счетчика.
* Записываемые управляющие поля (`POST /write`, подтверждено повторным чтением), соответствующие панели управления официальной интеграции, за исключением полей, помеченных в документации API как *зарезервированные*: заданное значение сети `GS`, максимальная подача электроэнергии `IS`, ограничения SoC `SI`/`SA`/`SO`, режим самопотребления `MM`, конфигурация счетчика `MD`, часовой пояс `TZ`, перезапуск `RT`, максимальная выработка сети `MG`, переключатели `LFB`/`LPS`/`PM` и локальный режим `LM` (⚠️ `LM=1` блокирует управление облаком/приложением до сброса). Зарезервированные поля (например, `PT`, `SI1`, `SA1`) доступны только для чтения.
* Два переключаемых **режима управления**: **контроллер самопотребления** на стороне адаптера (записывает `GS` из *любого* состояния счетчика ioBroker, с прямой связью + P, с сторожевым таймером/отказоустойчивостью), который **распределяет одну заданную точку сети между всеми датчиками**, или **саморегулирование устройства** (привязывает поддерживаемый счетчик к единому хранилищу и позволяет устройству управлять собой) — плюс **выключенный** режим для чистого мониторинга.
* Кнопка «Проверить все головки» в административной панели проверяет доступность каждой настроенной головки (модель + SoC) перед сохранением.
* Индикатор подключения (`info.connection`) плюс `info.lastUpdate` и `online` / `lastError` для каждого заголовка.
* Полный, неизмененный ответ `/read` каждого заголовка хранится в `heads.<n>.info.rawResponse` (JSON), поэтому любое поле, которое адаптер не сопоставляет с выделенным состоянием, все равно может быть прочитано оттуда.

## Как работает этот адаптер
Этот адаптер управляет хранилищем **локально**, без использования облачных сервисов производителя. Один экземпляр управляет **от одной до трех головок** (системами хранения данных). Самостоятельное использование может осуществляться **двумя взаимоисключающими способами** — вы выбираете один из них в настройке **Режим управления**:

**Режим B — Контроллер адаптера (рекомендуется по умолчанию, работает с любым счетчиком, 1–3 головки).** ioBroker считывает текущую мощность сети из **любого состояния, на которое вы его направите** (`gridPowerStateId`), а адаптер записывает заданное значение сети `GS` (прямая связь + коррекция P, с сторожевым таймером). Счетчик может быть *любым*, поддерживаемым ioBroker — Shelly, Tasmota, адаптером для интеллектуальных счетчиков/Modbus — **включая счетчики, которые хранилище не может считывать самостоятельно**. Вы указываете одно состояние, содержащее **чистую мощность сети в ваттах** (`>0` = потребление, `<0` = подача; включите *инвертирование знака источника*, если он перевернут; для счетчиков кВт / расщепленного импорта-экспорта / пофазных счетчиков сначала вычислите чистое значение сети в небольшом состоянии ioBroker). При наличии более чем одного датчика контроллер вычисляет **одно** общее заданное значение и **распределяет его между подключенными датчиками** — поровну, с ограничением по мощности каждого датчика, пропуская датчик, который полностью заряжен (во время зарядки) или разряжен (во время разрядки); его доля перераспределяется между остальными. Адаптер принудительно активирует `MM=0` для каждого датчика, чтобы устройства выполняли `GS`, и счетчик оставался полностью работоспособным в ioBroker.

**Режим A — Саморегулирование устройства (поддерживаемые счетчики, только с одним датчиком).** Адаптер привязывает поддерживаемый счетчик **к памяти** (`MM=1` + `MD`) и позволяет **устройству регулировать себя** — собственное потребление производителя, которое может реагировать быстрее, чем внешний контур. Этот режим доступен **только с одним датчиком**; при настройке двух или трех датчиков его нельзя выбрать — используйте вместо этого контроллер адаптера. Поддерживаются только четыре типа счетчиков (EcoTracker, Shelly 3EM, Shelly Pro 3EM, Tasmota), и счетчик должен быть доступен для памяти в локальной сети. В этом режиме адаптер **не** записывает `GS`. Привязка осуществляется просто путем опроса mDNS/HTTP, поэтому счетчик **остается доступным для использования в ioBroker** — в отличие от настроек счетчика в приложении производителя, которое может перенастроить счетчик и удалить его из ioBroker; этот адаптер привязывается напрямую и избегает этого.

**Выкл. (по умолчанию, только мониторинг).** Адаптер никогда не записывает состояния `MM`/`MD`/`GS`; он только опрашивает их. Вы по-прежнему можете вручную управлять состояниями `control.*`.

В обоих режимах управления адаптер **владеет `MM`**: при каждом опросе он проверяет `MM` каждого блока на соответствие выбранному режиму и повторно подтверждает его (с предупреждением), если что-то другое изменило его — таким образом, случайное подключение измерителя или внешний скрипт не могут незаметно отключить управление. Примечание: блок выполняет записанный `GS` только когда `MM=0`; при подключении измерителя (`MM=1`) он саморегулируется и игнорирует `GS`.

**Несколько блоков должны быть подключены к разным фазам.** Это ответственность оператора за электроснабжение — адаптер не может (и не должен) это проверять. Контроллер регулирует **суммарную (нетто)** мощность сети, отображаемую вашим счетчиком, — именно это учитывает стандартный немецкий двунаправленный счетчик с нетто-учетом; оптимизация по фазам выходит за рамки его компетенции.

**Требуется локальный режим (`LM=1`).** Каждое устройство предоставляет доступ к своему локальному HTTP API (`/read` / `/write`) только при включенном **локальном режиме** — при его выключении `/read` не возвращает данных (подтверждено на протестированной прошивке). Включение локального режима также отключает удаленное управление через облако/приложение; следовательно, мобильное приложение производителя больше не может управлять устройством.

## Требования
* В локальной сети могут быть доступны от одной до трех установок SunEnergyXT 500 (`PK=1`, 800 Вт) или 500 PRO (`PK=2`, 2400 Вт) (допустимы установки разных моделей).
* **Локальный режим (`LM=1`) включен** на каждом устройстве — это необходимо для передачи значений через локальный HTTP API (см. *Как работает этот адаптер*). Это также отключает удаленное управление из облака/приложения.
* В зависимости от режима управления требуется один счетчик: для **режима B** (контроллер адаптера) — любой счетчик, мощность сети которого доступна в качестве **состояния ioBroker**; для **режима A** (саморегулирование устройства, одноканальный) — один из четырех поддерживаемых счетчиков (EcoTracker, Shelly 3EM, Shelly Pro 3EM, Tasmota), доступных из хранилища в локальной сети. В режиме *выключено* не требуется.

## Установка
1. В административной панели ioBroker откройте раздел **Адаптеры**, найдите **sunenergyxt500** и установите его.
2. После установки будет создан экземпляр `sunenergyxt500.0`. Откройте его настройки и введите **IP-адрес / имя хоста Головного узла 1** (добавьте **Головной узел 2 / Головной узел 3**, если у вас больше головных узлов). Оставьте **Режим управления** в положении *Выкл* для чистого мониторинга.
3. Сохраните и закройте — адаптер начнет опрос и заполнит дерево объектов в каталоге `sunenergyxt500.0.heads.*` (и `total.*`).

## Конфигурация
**Связь**

* **IP-адрес/имя хоста головки 1** (обязательно) и **головка 2/головка 3** (необязательно) — локальные адреса ваших головок хранения, каждая с необязательной меткой. Одним экземпляром можно управлять до тремя головками. Разместите несколько головок на **разных фазах** (за это отвечает оператор); адаптер регулирует **суммарную** мощность сети. Один и тот же адрес нельзя ввести дважды.
* **Проверка всех головок** — проверяет каждую настроенную головку и сообщает модель + SoC (или ошибку), чтобы вы могли проверить адреса перед сохранением.
* **Интервал опроса (с)** — как часто каждая головка запрашивается через `/read` (по умолчанию 5 с).
* **Время ожидания запроса (мс)** — время ожидания HTTP-запроса (по умолчанию 8000 мс).

**Управление** — выберите **режим управления**:

*Выкл.* (по умолчанию) — только мониторинг; адаптер никогда не записывает `MM`/`MD`/`GS`.

*Контроллер адаптера* (Режим B) — поля:

* **Состояние источника питания от сети** — это состояние, в котором отображается потребляемая мощность вашего домашнего счетчика. Соглашение: `>0` = потребление от сети, `<0` = подача электроэнергии в сеть. Включите **Инвертировать знак источника**, если ваш счетчик использует противоположное соглашение.
* **Адаптивное управление** (по умолчанию включено): регулирует мощность в трех проверенных производителем режимах — небольшие отклонения плавно (каждые 7 с, шаг 20 Вт), средние — каждые 2,5 с (120 Вт), большие шаги нагрузки — немедленно (450 Вт), с фиксированной мертвой зоной 5 Вт. Отключите эту функцию, чтобы настроить контроллер вручную с помощью полей усиления / мертвой зоны / интервала записи / ограничения шага (они появятся только в этом случае).
* **Целевая мощность сети** (Вт, по умолчанию 0): 0 = нулевая подача электроэнергии в сеть; положительные значения обеспечивают небольшое преднамеренное потребление электроэнергии из сети (подача электроэнергии в сеть никогда не осуществляется), отрицательные значения — небольшую преднамеренную подачу электроэнергии в сеть — используется та же система знаков, что и для состояния источника («>0» = потребление).
* **Максимальная корректировка за одну коррекцию** (Вт, по умолчанию 500, 0 = неограниченно): ограничивает диапазон изменения заданного значения за один шаг управления, поэтому высокое усиление не может привести к перерегулированию при скачках показаний прибора.
* **Усиление** (по умолчанию 0,3), **Мертвая зона** (Вт), **Минимальный интервал записи** (мс), **Мертвая зона записи для каждой головки** (Вт — минимальное изменение заданного значения головки перед его перезаписью, чтобы избежать помех при смещении разделения). Максимальная мощность каждой головки **определяется автоматически** устройством (800 Вт для 500, 2400 Вт для 500 PRO), поэтому смешанные конфигурации работают без дополнительной настройки.
* **Предупреждение сторожевого таймера / отказоустойчивость** — если источник сигнала сети устаревает, контроллер регистрирует предупреждение и, наконец, принудительно устанавливает `GS=0` на **всех головках** (безопасное нейтральное положение) до восстановления источника. Телеметрия сторожевого таймера доступна в разделе `controller.*`.

Контроллер считывает фактическую мощность сети каждого устройства (`GP`) перед корректировкой, что обеспечивает естественную защиту от насыщения, когда устройство ограничивает мощность внутри себя (например, с помощью SoC).

*Саморегулирование устройства* (Режим A, **только для одной головки**) — поля:

* **Тип счетчика** — EcoTracker / Shelly 3EM / Shelly Pro 3EM / Tasmota.
* **Серийный номер / IP-адрес счетчика** — серийный номер для Shelly/Tasmota (определяется через mDNS) или IP-адрес локальной сети для EcoTracker (напрямую). Для Tasmota используйте префикс серийного номера без последних 4 символов и установите **кнопку питания** в соответствии с подтипом вашего энергомонитора.

Адаптер привязывает счетчик (`MM=1` + `MD`), и устройство саморегулируется; адаптер не записывает `GS`. Привязанный счетчик остается доступным для использования в ioBroker. Этот режим скрывается/блокируется после настройки второго или третьего датчика.

> **Безопасность:** В режиме *Выкл* адаптер работает только в режиме чтения — он только опрашивает состояние `/read` и никогда не записывает данные, если вы не зададите состояние `control.*`. В режиме управления адаптер **принуждает к состоянию `MM`** для каждой головки в этом режиме и подтверждает его, если оно изменено извне; **не** запускайте одновременно второй скрипт записи `GS` (ваш собственный скрипт или скрипт `MM` устройства с другим индикатором), иначе они будут конкурировать за заряд батареи.

## Управление поведением, точностью и ограничениями
**Чего ожидать:** Контроллер поддерживает мощность сети в диапазоне, обычно составляющем **±10–20 Вт вокруг нуля**, и корректирует скачки нагрузки — в зависимости от настроек — в течение **~1–3 секунд до ~30 секунд**. Постоянное, точное значение 0,0 Вт **не достижимо по замыслу** — с любым контроллером на этом оборудовании:

* **Точность и уровень шума измерителя:** сам внешний измеритель измеряет с погрешностью и уровнем шума в несколько ватт — регулирование ниже этого значения бессмысленно. (Уставка `GS` имеет разрешение 1 Вт, поэтому точность установки не является пределом.)
* **Задержка в цепочке измерений:** измерение → состояние ioBroker → контроллер → `/write` → устройство изменяет параметры. Между этапом загрузки и его коррекцией неизбежно проходит ~1–3 секунды.
* **Динамика нагрузки:** Компрессор или чайник переключаются за миллисекунды — после этого реагирует каждый контроллер. Кратковременные скачки мощности на ваших графиках являются нормальными и энергетически незначимыми (ватт-секунды).
* Контроллер регулирует мощность в соответствии с заданной **целевой мощностью сети** (по умолчанию 0) и **симметрично** колеблется вокруг нее — короткие, небольшие моменты подачи электроэнергии являются частью нулевой подачи электроэнергии. Если подача электроэнергии никогда не требуется, установите целевое значение на небольшое преднамеренное потребление (например, +10 Вт).

**Как работают настройки:**

| Настройка | Эффект | меньшее значение | большее значение |
|---|---|---|---|
| **Усиление** | доля отклонения, корректируемая за шаг | медленный, плавный (0,3 ≈ 7 шагов до ~0) | быстрый (1,0 = одна коррекция), но сильнее реагирует на шум измерительного прибора; >1 может привести к перерегулированию |
| **Целевая мощность сети (Вт)** | значение, до которого регулируется мощность сети | <0 = преднамеренное включение подачи электроэнергии в сеть | >0 = преднамеренное потребление электроэнергии («никогда не включать подачу электроэнергии в сеть») |
| **Максимальная регулировка на одну коррекцию (Вт)** | ограничивает изменение заданного значения за шаг | сглаживает скачки показаний прибора при высоком коэффициенте усиления | большее значение (или 0 = неограниченное) быстрее реагирует на большие скачки нагрузки |
| **Мёртвая зона (W)** | отклонения ниже неё игнорируются | более точная, больше записей (0 = всё исправлено) | более спокойная, оставляет небольшое постоянное отклонение |
| **Минимальный интервал записи** | частота корректировок | более быстрое стабилизирование (минимальное значение 1000 мс) | меньше операций записи на устройство, более медленное отслеживание |
| **Мёртвая зона записи для каждой головки** | подавляет мини-перераспределения между головками (многоголовочная обработка) | более точная | меньше помех |

**Адаптивное управление** (по умолчанию) автоматически выбирает темп для каждого уровня. Для ручного режима доступны два проверенных профиля: *Расслабленный* (по умолчанию — спокойный режим, минимальное количество операций записи на устройство, диапазон ±20–30 Вт) и *Точный* (усиление 0,8–1,0 · мертвая зона 0 · интервал 1000 мс · мертвая зона записи 0 — диапазон ±10–20 Вт, стабилизация за 1–3 с). В течение суток оба профиля дают практически одинаковый энергетический баланс — разница заключается в визуальном оформлении графика, а не в стоимости.

## Правила оформления знаков
* `GP` (мощность сети): `>0` = подача электроэнергии в сеть, `<0` = потребление — **противоположно показаниям счетчика Shelly** (`api.GP ≈ −shelly.gridPower`).
* `BP` (мощность батареи): `>0` = зарядка, `<0` = разрядка.
* `GS` (уставка для сети): `>0` = подача/разрядка электроэнергии, `<0` = зарядка от сети (±2400 Вт на модели Pro, разрешение 1 Вт).

## Дерево объектов
Каждый блок получает собственное поддерево в рамках **`heads.<n>.*`** (`n` = 1…3), а также объединенные агрегаты **`total.*`** и адаптеры `controller.*` / `info.*`. Внутри блока состояния группируются в тематические каналы; **листом каждого идентификатора объекта является код поля API устройства** (идентификатор сущности из официального справочника полей), а двуязычное имя объекта описывает его — таким образом, дерево соответствует документированным полям устройства в соотношении 1:1.

| Канал | Содержание |
|---|---|
| `heads.<n>.battery.*` | SoC (`SC`), мощность батареи (`BP`), SoC на один блок (`SC0`–`SC5`), количество блоков в сети (`ON`), гистерезис SoC (`SI1`/`SA1`) |
| `heads.<n>.load.*` | мощность нагрузки (`LP`), суточная энергия автономной нагрузки (`LD`) |
| `heads.<n>.pv.*` | общая мощность фотоэлектрических систем (`PV`) и мощность/ток/напряжение на каждый MPPT (`mppt1`–`mppt4`) |
| `heads.<n>.system.*` | общая входная/выходная мощность (`IW`/`OP`) |
| `heads.<n>.device.*` | тип/модель/серийный номер/статус; `network.*` (IP, порт, Wi-Fi); `firmware.*` (`ES`/`AS`/`DS` программное обеспечение, `EH`/`AH`/`DH` аппаратное обеспечение, `BS0`–`BS5` BMS) |
| `heads.<n>.meter.*` | состояние внешнего счетчика (`MS`) |
| `heads.<n>.ups.*` | Режим ИБП / зарядка от сети / обходной режим (`UO`/`UG`/`FP`) |
| `heads.<n>.fault.*` | Битовые маски ошибок (`TF`/`EF`/`DF1`/`DF2`/`AF1`/`AF2`/`BF`) — заполняются только во время активной ошибки |
| `heads.<n>.control.*` | все **доступные для записи** поля (см. ниже) |
| `heads.<n>.info.*` | на человека `online`, `lastError`, `rawResponse` (полный исходный текст `/read`) |
| `total.*` | объединенный вид: взвешенный по пропускной способности `soc`, суммированный `batteryPower` / `gridPower` / `maxPower`, `onlineCount` |
| `controller.*` | телеметрия контроллера собственного потребления (`status`, возраст источника энергии в сети) |
| `info.*` | `connection` (любая доступная голова) и `lastUpdate` |
| `info.*` | `connection` (любой доступный головной узел) и `lastUpdate` |

### Записываемые элементы управления (`heads.<n>.control.*`)
Согласно соглашению ioBroker, все поля, доступные для записи, находятся в разделе `control.*` для каждого заголовка. Поскольку это упрощает их структуру, в этой таблице показано, к чему относится каждый из них:

| Объект | Поле | Относится к | Описание |
|---|---|---|---|
| `control.GS` | GS | сеть | Уставка мощности сети (`>0` подача электроэнергии / `<0` заряд сети) |
| `control.MG` | МГ | сеть | Макс. выходная мощность, подключенная к сети |
| `control.SI` | SI | батарея | Минимальный уровень заряда батареи (сетевой режим) |
| `control.SA` | SA | батарея | Максимальный уровень заряда (сетевой режим) |
| `control.SO` | SO | батарея | Минимальный уровень разряда SoC (автономный режим) |
| `control.MM` | MM | режим | Локальный режим нулевой подачи электроэнергии / самопотребления (в сочетании с `MD`) |
| `control.MD` | MD | счетчик | JSON-файл подключения счетчика (в сочетании с `MM`) |
| `control.LM` | LM | режим | Локальный режим (⚠️ `1` блокирует управление облаком/приложением) |
| `control.LFB` | LFB | режим | Переключатель приоритета нагрузки |
| `control.LPS` | LPS | режим | Автономный выходной переключатель |
| `control.PM` | PM | режим | Параллельный режим |
| `control.TZ` | TZ | устройство | POSIX часовой пояс |
| `control.RT` | RT | устройство | Перезагрузить устройство (кнопка — мягкая перезагрузка, **не** полное выключение и включение питания) |
| `control.RT` | RT | устройство | Перезагрузить устройство (кнопка — мягкая перезагрузка, **не** полное выключение и включение питания) |

> Совет: в административной панели ioBroker вы также можете отфильтровать список объектов по флагу *доступен для записи*, чтобы найти все элементы управления сразу.

`device.PK` является производным от `DevType` в прошивке, которая больше не сообщает о `PK`. Зарезервированные поля (`PT`, `SI1`, `SA1`) доступны только для чтения. Поля, которые производитель удалил (`PD`, `UP`) или которые являются артефактами только для документации (`WT`, `BN`), не доступны; все не отображенные поля по-прежнему доступны в `heads.<n>.info.rawResponse`.

## Поля ручного ввода показаний счетчика/режима (MM / MD)
`MM`/`MD` — это собственное потребление энергии головкой на основе показаний счетчика. При выборе **режима управления** адаптер управляет ими автоматически (режим A устанавливает `MM=1` + `MD` для одной головки; режим B принудительно устанавливает `MM=0` для каждой головки), а его защитный механизм повторно устанавливает соответствующий режиму `MM` при следующем опросе — поэтому любое ручное изменение режима управления является временным.

Исходные поля остаются доступными для записи при использовании экспертами/вручную (например, в режиме *Выкл*). Они соответствуют официальной схеме сопряжения: выключение `MM` также очищает `MD`, а запись `MD` включает `MM` (непустое значение) или отключает его (пустое значение). JSON-форматы `MD` для четырех поддерживаемых счетчиков находятся в локальной справочной информации API устройства; в режиме *Саморегулирование устройства* адаптер формирует их автоматически на основе типа счетчика и серийного номера/IP-адреса.

## Ограничения
* **До трех головок на один экземпляр.** Работа с одной головкой проверена на реальном оборудовании; разделение на несколько головок проверено модульными тестами, но на момент написания статьи **не тестировалось на реальной установке с 2–3 головками** — отзывы от пользователей с многоголовочными системами приветствуются. *Саморегулирование устройства* доступно только для одной головки.
* **Электростанции должны быть подключены к разным фазам** (ответственность оператора). Адаптер регулирует **суммарную** мощность сети, а не мощность каждой фазы отдельно.
* Балансировка заряда каждого отдельного аккумулятора осуществляется собственной системой управления батареей (BMS) каждого устройства — адаптер управляет только общей мощностью устройства и считывает значение `battery.SC` (total) для контроля; он не управляет отдельными аккумуляторами.
* Ежедневные счетчики энергии (`GD1`/`GD2`/`LD`) показывают необработанные значения **Вт·ч**, а не кВт·ч.
* `MD` и `TZ` вступают в силу немедленно, но не гарантируется их точное воспроизведение устройством — подтвердите действие, а не воспроизведение.
* **Входные сигналы PV не тестировались с оборудованием** (эталонная установка работает без солнечных модулей, поэтому `PV1–4` всегда равны 0). Интеграция и контроллер не зависят от типа солнечных панелей и являются полными, но непроверенные сценарии работы прошивки PV (например, полная батарея + избыток солнечной энергии, поля ИБП/байпаса `FP`/`UG`) не проверены — обратная связь приветствуется.

## Поиск неисправностей
* **`info.connection` остается `false` / нет данных:** сначала убедитесь, что **локальный режим (`LM=1`)** включен на устройстве — без него локальный API не возвращает значений. Затем проверьте, доступен ли `http://<device-ip>/read` с хоста ioBroker (проверьте с помощью браузера или `curl`). Для каждого заголовка `heads.<n>.info.online` и `heads.<n>.info.lastError` показывают, какой из них завершился с ошибкой.
* **Ничего не контролируется:** проверьте **режим управления** — *Выкл* никогда не записывается. В *контроллере адаптера* установите допустимое **состояние источника питания от сети**; в *саморегулировании устройства* установите поддерживаемый **тип счетчика** и **SN/IP**.
* **Устройство игнорирует `GS` / батарея не реагирует:** головка выполняет записанный `GS` только тогда, когда `MM=0`. В режиме *контроллера адаптера* адаптер обеспечивает это; если вы записываете `GS` вручную, убедитесь, что ни один измеритель не привязан (`MM=0`). При привязанном измерителе (`MM=1`) устройство саморегулируется и игнорирует `GS`.
* **Контроллер работает слишком медленно / никогда не достигает точно 0:** см. *Поведение управления, точность и ограничения* — измерительная цепочка добавляет задержку примерно в 1–3 секунды, а сам измеритель имеет конечную точность, поэтому физическим оптимумом является диапазон ±10–20 Вт вокруг целевого значения. Для максимально быстрой реакции используйте профиль *Точный* (усиление 0,8–1,0, мертвая зона 0, минимальный интервал записи 1000 мс); чтобы никогда не подавать сигнал, установите **целевую мощность сети** на небольшое положительное значение.
* **Временные метки состояний выглядят устаревшими / флаг качества 32:** адаптер записывает состояние только тогда, когда изменяется его значение (стандартная практика — это защищает базу данных состояний от миллионов одинаковых записей). Поэтому временная метка состояния показывает последнее *изменение* значения, а не последний опрос. Проверьте актуальность данных с помощью `info.lastUpdate` (обновляется при каждом успешном опросе) и `heads.<n>.info.online`. Качество 32 («замена исходного значения») сохраняется только для состояний, которые устройство никогда не предоставляет (например, SoC пакетов расширения, которые не установлены); после каждого запуска адаптера все предоставленные значения записываются один раз, поэтому их временные метки как минимум такие же актуальные, как и при запуске.
* **Два контроллера борются за батарею:** запускайте только один. Адаптер принудительно устанавливает `MM` для выбранного режима — отключите любой внешний скрипт `GS` (или собственный `MM` устройства с другим измерителем) перед использованием режима управления.
* **Некоторые состояния остаются пустыми (`0` / `""`):** устройство возвращает только те поля, которые фактически предоставляет его прошивка/топология (например, дополнительные пакеты `SC2`–`SC5` или битовые маски ошибок только во время ошибки). Полный необработанный ответ всегда доступен в `heads.<n>.info.rawResponse`.
* **После обновления с одноголовочной версии дерево выглядит неправильно:** в версии 0.2.0 дерево объектов было перестроено в `heads.<n>.*`. Адаптер автоматически удаляет устаревшие объекты при запуске; если что-то осталось, удалите старые объекты (или добавьте экземпляр заново).
