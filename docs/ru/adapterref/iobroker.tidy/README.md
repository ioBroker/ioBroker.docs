# Адаптер ioBroker для обнаружения неиспользуемых состояний
---

## Что делает этот адаптер
Адаптер **Tidy** помогает находить неиспользуемые объекты и состояния для очистки вашей системы. После многих лет использования ioBroker в вашей системе могут накапливаться «мертвые» точки данных из-за удаленных скриптов, удаленных адаптеров или заброшенных конфигураций. Этот адаптер помогает идентифицировать и очищать такие точки данных, чтобы ваша система оставалась чистой и производительной.

## Функции
- **📊 Сканирование по пути**: Настройте несколько путей для сканирования с помощью средства выбора объектов (например, `0_userdata.0`, `alias.0`)
- **🚫 Исключения**: Исключить из результатов сканирования намеренно статичные точки данных, целые папки или шаблоны с подстановочными знаками.
- **🔍 Интеллектуальное обнаружение**: Выявляет различные типы проблемных точек данных:
- **Неактивный**: Никогда не обновлялся или очень старый (настраиваемый порог, по умолчанию: 365 дней)
- **Устаревшая информация**: Не обновлялась в последнее время (настраиваемый порог, по умолчанию: 90 дней)
- **Осиротевшие псевдонимы**: Псевдонимы, указывающие на несуществующие целевые точки данных.
- **Неопределено**: Точки данных, в которые не было произведено ни одной записи с момента запуска системы.
- **⚙️ Гибкая конфигурация**:
- Настраиваемые пороговые значения возраста для обнаружения "недействительных" и "мертвых" данных.
- Дополнительное автоматическое сканирование с настраиваемыми интервалами (ежечасно)
- Включение/отключение отдельных путей сканирования
- Дополнительно: полное сканирование всего дерева объектов.
- **🌐 Полное сканирование**: сканирование всех состояний в системе (не ограничиваясь настроенными путями) через отдельный канал результатов.
- **🎯 Ручные триггеры**: Для каждого настроенного пути предусмотрена кнопка запуска сканирования по запросу.
- **📋 Вывод в формате JSON-таблицы**: Результаты предоставляются в виде массивов JSON, идеально подходящих для табличных виджетов в VIS.
- **📈 Статистика**: Автоматические счетчики общего количества, неактивных, устаревших и потерянных данных

## Конфигурация
### Общие настройки
- **Включить автоматическое сканирование**: При включении этой функции все настроенные пути будут сканироваться автоматически.
- **Сканировать все объекты (полностью)**: При включении этой функции адаптер сканирует все дерево объектов и сохраняет результаты в отдельном канале `complete` (см. ниже).
- **Интервал сканирования**: Как часто должно выполняться автоматическое сканирование (в часах, минимум 1)
- **Количество дней до "устаревших" данных**: Данные, не обновлявшиеся в течение указанного количества дней, помечаются как устаревшие (предупреждение)
- **Количество дней до «неактивности»**: Точки данных, не обновлявшиеся в течение указанного количества дней, помечаются как неактивные (критические).

### Пути сканирования
Настройте один или несколько путей для мониторинга:

- **Включено**: Включить/отключить этот путь сканирования
- **Путь сканирования**: выберите папку из дерева объектов или введите путь вручную (например, `0_userdata.0`, `alias.0`, `javascript.0`). Поле остается редактируемым после выбора.
- **Все экземпляры адаптера**: Удалите номер экземпляра, чтобы просканировать каждый экземпляр — например, `alias` вместо `alias.0` соответствует `alias.0`, `alias.1` и так далее.
- **Имя**: Удобное для восприятия имя для этого пути (используется для именования состояний результата)
- **Проверка псевдонимов целей**: Для путей `alias.*` проверьте, существуют ли еще целевые точки данных (обнаружение "фантомов")

### Исключения
Исключите из результатов сканирования или статистики точки данных, которые не должны отображаться:

- **Объект**: Выберите одно состояние или папку/канал, чтобы исключить все поддерево. Поддерживаются подстановочные знаки: `*` соответствует любому суффиксу, `?` соответствует одному символу (например, `0_userdata.0.rollo.trigger*`).
- **Комментарий**: Необязательное примечание (например, почему этот показатель исключен)

Используйте это для параметров конфигурации, которые редко меняются (например, сохраненная радиостанция для сценария оповещения), чтобы они не помечались как устаревшие или неактуальные.

Исключения касаются сканирования по траектории и полного сканирования. Каждый канал сканирования предоставляет **`exceptionCount`** — количество точек данных, исключенных во время последнего сканирования (0, если ни одной).

## Точки данных
Для каждого заданного пути (например, "userdata") адаптер создает:

- **`tidy.0.userdata.trigger`** (кнопка): Нажмите, чтобы запустить сканирование вручную.
- **`tidy.0.userdata.result`** (json): Полные результаты сканирования в виде таблицы JSON
- **`tidy.0.userdata.lastScan`** (временная метка): Время выполнения последнего сканирования
- **`tidy.0.userdata.count`** (число): Всего найдено точек данных
- **`tidy.0.userdata.deadCount`** (число): Количество неработающих точек данных
- **`tidy.0.userdata.staleCount`** (число): Количество устаревших точек данных
- **`tidy.0.userdata.orphanedCount`** (число): Количество осиротевших псевдонимов
- **`tidy.0.userdata.exceptionCount`** (число): Количество точек данных, исключенных из списка исключений

Если в общих настройках включена опция **Сканировать все объекты (полностью)**, адаптер дополнительно создает:

- **`tidy.0.complete.trigger`** (кнопка): Нажмите, чтобы вручную запустить полное сканирование
- **`tidy.0.complete.result`** (json): Результаты сканирования для всех состояний в системе
- **`tidy.0.complete.lastScan`** (временная метка): Время выполнения последнего полного сканирования
- **`tidy.0.complete.count`** (число): Всего найдено точек данных
- **`tidy.0.complete.deadCount`** (число): Количество неработающих точек данных
- **`tidy.0.complete.staleCount`** (число): Количество устаревших точек данных
- **`tidy.0.complete.orphanedCount`** (число): Количество осиротевших псевдонимов
- **`tidy.0.complete.exceptionCount`** (число): Количество точек данных, исключенных из списка исключений

Полное сканирование использует ту же структуру результатов в формате JSON, что и сканирование по путям. При включении этой опции автоматическое сканирование включает в себя полное сканирование.

### Структура результатов JSON
Состояние `result` содержит JSON-массив со следующими полями для каждой точки данных:

```json
[
  {
    "id": "0_userdata.0.hallway.light_auto",
    "name": "Hallway Light Automation",
    "last_ts": 1712856000000,
    "last_ts_iso": "2026-04-11T16:00:00.000Z",
    "value": true,
    "status": "active",
    "issue": null,
    "size": 4
  },
  {
    "id": "0_userdata.0.test.old_value",
    "name": "Test Datapoint",
    "last_ts": null,
    "last_ts_iso": "undefined",
    "value": 15,
    "status": "undefined",
    "issue": "dead",
    "size": 2
  }
]
```

**Описание полей:**

| Область | Описание | Назначение |
|-------|-------------|---------|
| `id` | Полный путь к точке данных | Уникальный идентификатор |
| `last_ts` | Временная метка Unix (мс) или null | Сортировка в фоновом режиме |
| `last_ts_iso` | Строка даты ISO 8601 | Отобразить в таблице |
| `value` | Текущее значение точки данных | Окончательная проверка перед удалением |
| `status` | `active`, `dead`, `stale`, `undefined`, `orphaned` | Классификация (английский) |
| `status_de` | `aktiv`, `inaktiv`, `veraltet`, `undefiniert`, `verwaist` | Классификация (немецкий) |
| `issue` | `dead`, `stale`, `orphaned_alias` или `null` | Критерий фильтрации (null = OK) |
| `issue_de` | `inaktiv`, `veraltet`, `verwaistes Alias` или `null` | Критерий фильтрации (немецкий) |
| `size` | `JSON.stringify(val).length` | Обнаруживает "пожирателей складских запасов" |
| `size` | `JSON.stringify(val).length` | Находит "пожирателей памяти" |

## Примеры использования
### Базовая настройка
1. Установите и настройте адаптер.
2. Добавьте путь для сканирования с помощью средства выбора объектов (например, `0_userdata.0`) или введите его вручную.
3. Дайте ему имя (например, "userdata")
4. Сохраните конфигурацию.
5. Адаптер немедленно выполнит первоначальное сканирование.
6. Просмотрите результаты в `tidy.0.userdata.result`

### Пути сканирования нескольких экземпляров
Чтобы просканировать все экземпляры адаптера, удалите номер экземпляра из пути после выбора:

- `alias.0` → сканировать только первый экземпляр псевдонима
- `alias` → сканирование `alias.0`, `alias.1` и любых других экземпляров псевдонимов

Поле "Путь" остается полностью редактируемым после использования средства выбора объекта.

### Исключения
1. Откройте вкладку **Исключения** в конфигурации адаптера.
2. Добавьте данные, которые следует игнорировать (например, редко обновляемые значения конфигурации).
3. Выберите одно состояние для одной точки данных, папку для исключения целого поддерева или введите шаблон с подстановочным знаком (например, `0_userdata.0.rollo.trigger*`).
4. Сохраните конфигурацию — исключенные точки данных больше не будут отображаться в результатах сканирования или подсчетах.

### Интеграция VIS
Используйте результат в формате JSON с табличным виджетом для отображения и сортировки ваших данных:

1. Создайте табличный виджет в VIS (например, inventtwo Table Widget).
2. Привяжите его к `tidy.0.userdata.result`
3. Настройка столбцов:
- Для таблиц на **немецком** языке: `id`, `name`, `last_ts_iso`, `status_de`, `issue_de`
- Для **англоязычных** таблиц: `id`, `name`, `last_ts_iso`, `status`, `issue`
4. Отсортируйте по `last_ts` (начиная с самых старых данных), чтобы найти самые "устаревшие" точки данных.
5. Отфильтруйте данные по критерию `issue != null`, чтобы отобразить только проблемные точки данных.

### Полное сканирование
1. В общих настройках включите параметр **Сканировать все объекты (полностью)**.
2. Сохранение конфигурации — адаптер выполняет первоначальное полное сканирование.
3. Результаты можно посмотреть в файле `tidy.0.complete.result`.
4. Используйте `tidy.0.complete.trigger` для ручного повторного сканирования в любое время.

Используйте полное сканирование, чтобы получить обзор всех состояний вашего экземпляра. Для целенаправленной очистки обычно более практичны сканирования по путям (например, `0_userdata.0`, `alias.0`).

### Автоматическое обслуживание
1. Включите функцию "автоматическое сканирование" в настройках.
2. Установите интервал в 24 часа (один раз в день).
3. Отслеживайте статистику `deadCount` и `staleCount`.
4. Еженедельно анализируйте результаты, чтобы определить кандидатов на очистку.
